If you're a Medicare broker running a voice AI agent for needs assessments, you have one of the most regulated voice AI use cases that exists. You're sitting at the intersection of three regulatory regimes simultaneously:

1. **HIPAA** — you're a Business Associate of the health plans you sell on behalf of
2. **CMS chain-of-enrollment rules** — every needs assessment, marketing, and enrollment call must be recorded and retained for 10 years, with TPMO disclaimers and explicit recording consent
3. **State two-party consent laws** — recording without consent is a felony in CA, FL, IL, MA, MD, MT, NH, NV, PA, and WA

And you're running this on Vapi, which means PHI is flowing through at least four vendors per call: Vapi (orchestration), Deepgram (STT), OpenAI or Anthropic (LLM), and ElevenLabs (TTS). Plus the recording sits somewhere.

This post walks the full architecture: what gets captured, where PHI enters and exits, how to keep your LLM context clean, and how to produce per-call audit artifacts that survive a CMS audit. It also shows where Masker fits in.

## The compliance stack you actually have to handle

A Medicare needs assessment call has to satisfy all of these simultaneously:

**HIPAA Security Rule (technical safeguards on PHI):**

- Encryption in transit (TLS 1.2+) and at rest (AES-256)
- Unique user identification, MFA, role-based access
- Immutable audit controls — who accessed what ePHI, when, retained 6 years
- Integrity controls and automatic logoff

**HIPAA Privacy Rule (use of PHI):**

- Minimum necessary — your LLM doesn't need a beneficiary's name to answer "does plan X cover Eliquis"
- Permitted uses constrained by your BAA with the health plan

**CMS 2026 chain-of-enrollment recording mandate ([source](https://www.thunai.ai/blog/cms-call-recording-requirements)):**

- Record the entire call from before the first greeting through hangup
- TPMO disclaimer within the first 60 seconds
- Verbal consent to record before any PHI capture
- Verification of agent identity and non-affiliation with the federal government
- Audio retained 3 years, transcript +3 years, enrollment records 10 years
- Penalties up to $1,000 per record per day; potential revocation of Medicare certification

**TCPA + state consent rules:**

- Two-party consent in 10+ states
- Recording consent must precede any PHI capture

What this means architecturally: you must record everything, retain everything, and *also* keep raw PHI out of the systems that don't need it.

## What flows where on a default Vapi-based broker call

Here's the data flow on a typical Vapi-based Medicare needs assessment, with PHI exposure flagged at each hop:

```
[ Beneficiary phone ]
        │  PSTN audio — full PHI
        ▼
[ Twilio / SIP carrier ]
        │  TLS, but full audio with PHI
        ▼
[ Vapi orchestration ]  ◄── recording captured here
        │
        ├──► [ Deepgram STT ] ──── PHI in transcript
        │
        ├──► [ OpenAI / Anthropic LLM ] ◄── full prompt with PHI
        │              │
        │              ▼
        │       LLM response
        │
        ├──► [ ElevenLabs TTS ] ──── synthesizes response audio
        │
        └──► [ Your CRM ] ◄── transcript + extracted data,
                              long-term storage with raw PHI

[ S3 / Vapi storage ] ◄── full call recording, 10-year retention
```

The exposures hidden in this flow:

1. **Every LLM prompt contains the full conversation history** — beneficiary name, DOB, MBI, medications, doctor names, address — even though the LLM only needs the medical and plan-relevant facts.
2. **Transcripts land in your database in plaintext**, accessible to anyone with database access.
3. **Recordings live in S3 or Vapi storage** for 10 years — required by CMS — but are commonly accessible via presigned URLs to engineers and QA.
4. **Subcontractor BAAs are required** for every vendor that processes PHI on your behalf. [Anthropic requires the Enterprise plan for a BAA](https://privacy.claude.com/en/articles/8114513-business-associate-agreements-baa-for-commercial-customers); [OpenAI requires sales contact](https://community.openai.com/t/hipaa-business-associates-agreement/118007); Vapi charges $0.015/min or $1,000/month for HIPAA-tier hosting.
5. **The CMS audit trail is incomplete.** You have the recording, but you don't have a structured per-call artifact saying: TPMO disclaimer delivered at 00:00:14, recording consent obtained at 00:00:42, agent identified as licensed at 00:00:51, PHI categories captured and redacted before LLM, hash X.

That last gap is the one that bites you in a CMS audit.

## What "minimum necessary" looks like for a Medicare LLM

HIPAA's [minimum necessary standard](https://www.hhs.gov/hipaa/for-professionals/privacy/guidance/minimum-necessary-requirement/index.html) says you should only use the PHI you need for the task. For a Medicare needs assessment LLM:

| Field | Does the LLM need it? | Send to LLM? |
| --- | --- | --- |
| Beneficiary name | No | Replace with token |
| Date of birth | Year only is enough for plan eligibility | Year only |
| Medicare Beneficiary ID (MBI) | No — used only by enrollment systems | Replace with token |
| Address | ZIP only for plan availability | ZIP only |
| Current medications | Yes — Part D formulary lookup | Send drug names |
| Current PCPs / specialists | Yes — provider network check | Send provider names + NPI |
| Current plan name | Yes — comparison logic | Send |
| Phone number | No — Vapi already has it for telephony | Replace with token |
| SSN / SSN tail | Never. | Block entirely |
| Diagnoses | Sometimes — for plan recommendation | ICD-10 code; redact narrative |

Roughly half the PHI flowing into your LLM today doesn't need to be there. Removing it doesn't change the agent's quality — it changes your blast radius if anything goes wrong.

## How Masker fits in

Masker sits between Vapi and your LLM as a drop-in OpenAI-compatible proxy. You change one URL in Vapi's "custom LLM" field — from `api.openai.com` to your per-agent Masker proxy — and PHI is tokenized before it leaves your trust boundary. The recording stays intact for CMS retention; the LLM never sees raw PHI.

```
[ Vapi orchestration ]
        │
        │  /proxy/{agent_id}/v1/chat/completions
        ▼
[  M A S K E R  ] ──► [ Encrypted recording vault ]
        │                  full PHI, 10-year retention,
        │                   immutable, audit-logged access
        │
        │  tokenized prompt ──► [ LLM (OpenAI / Anthropic) ]
        │  rehydrated response ◄──┘
        │
        ├──► [ Redacted transcript store ] for ops/QA
        │
        ├──► [ Per-call compliance report ] CMS + HIPAA artifact
        │
        └──► [ Audit event stream ] HMAC-chained, append-only
```

What that looks like in practice for a Medicare broker:

### 1. Two-pass detection tuned for Medicare

Masker's [detection pipeline](/masking/detection) runs two passes on every payload:

- **Pass 1 — regex catalogue:** sub-millisecond. Phones, SSNs, MRNs, dates, ZIPs, account numbers, plus tenant-configurable patterns (Medicare Beneficiary IDs, Part D plan IDs, contract numbers, NPIs).
- **Pass 2 — Gemma-4 NER:** 30–80 ms. Catches names, geographic subdivisions, payer-specific beneficiary numbers, and numbers spoken as words ("five five five, one two three..."). Fine-tuned on i2b2/n2c2, MedDialog, and Switchboard.

For Medicare specifically, you can extend Pass 1 with a per-agent policy that includes MBI format validation, plan contract numbers (H####, R####, S####, E####), and HICN (legacy). See [Mask policy](/configuration/mask-policy) for how to register custom patterns.

### 2. Reversible tokens, not surrogates

Detected entities are replaced with [versioned, type-aware tokens](/masking/tokenization) like:

```
MSKV1.PHONE.K_HEALTHCARE.A1B2C3D4E5F6G7H8I9J0KL
```

The token is format-stable (the LLM still recognizes it as a phone-shaped argument), reversible by Masker on the rehydration leg (so the agent can say "Margaret" out loud in the TTS response), and non-revealing if leaked. Two schemes are available: vault-deterministic (HMAC-SHA256, token consistency across calls) and AEAD (authenticated encryption, no shared vault required).

If your LLM provider has a logging incident, the leaked transcripts contain `MSKV1.NAME.K_HEALTHCARE.…`, not Margaret Thompson.

### 3. Drop-in Vapi integration

Two endpoints, both webhook-style:

- **[Custom LLM endpoint](/api-reference/vapi/proxy-chat-completions)** — `POST /proxy/{agent_id}/v1/chat/completions`. OpenAI-compatible. You put it in Vapi's "custom LLM" field and Masker handles masking + forwarding + rehydration on the streaming response.
- **[Assistant request webhook](/api-reference/vapi/assistant-request-webhook)** — `POST /vapi/webhook/{agent_id}`. Vapi-specific; returns the assistant configuration for each call and attaches audit metadata.

Latency overhead on the chat completion path is **45–95 ms** (see [Voice pipeline](/masking/voice-pipeline)) — typically 5–10% of the 800–1,200 ms end-to-end voice budget. In pilots, blind A/B listeners haven't been able to detect it.

### 4. CMS chain-of-enrollment compliance detection

The part most teams overlook. Masker analyzes each call and produces structured evidence that the CMS-required compliance moments happened:

- **TPMO disclaimer detected** in the first 60 seconds, with timestamp and quote
- **Recording consent verified** — explicit verbal "yes," with timestamp
- **Agent identification + non-affiliation** confirmed — agent identified as licensed, clarified non-affiliation with the federal government / Medicare
- **Scope of Appointment drift** flagged — if a Med Supp call drifts into MA/PDP territory without an SOA on file
- **Superlative usage** flagged — CMS-discouraged language ("best plan," "top-rated") without backing data

Each compliance event is a row in the per-call report with a timestamp and a verbatim quote.

### 5. Per-call signed audit reports

For every session, Masker generates an [Ed25519-signed compliance report](/compliance/audit-reports) (PDF, self-contained) with:

- Cover: scope, time window, generation timestamp, root hash
- Executive summary: total redactions, coverage %, alert count
- HIPAA Safe Harbor matrix at the time of generation (see [coverage](/compliance/hipaa-safe-harbor))
- Redaction summary by entity kind and detection pass
- Sample masked transcripts (random sample)
- Audit log integrity: start hash, end hash, event count, verification status
- Configuration snapshot: active policy YAML and per-agent upstream
- Appendix: list of every session in scope (IDs only, no PHI)

The audit log itself is **HMAC-SHA256 chained** — `event_n.hash = HMAC(audit_key, event_{n-1}.hash || canonical_json(event_n))`. Tampering with any event invalidates every subsequent hash. This is the artifact you hand to CMS in an audit, or to a health plan partner during their annual TPMO oversight review.

### 6. Redacted transcript for ops use

Your QA team, sales coaches, and analytics pipeline don't need raw PHI. Masker stores a redacted version of every transcript that's safe to query, search, and use for training data. The full PHI version stays in the encrypted vault, accessible only via audit-logged, MFA-gated access for genuinely necessary purposes.

## What changes for the broker

With this stack:

- LLM context contains zero direct identifiers — minimum necessary, by construction
- Your transcript database contains redacted text — your team can review calls without an HR-flag-worthy PHI exposure every time
- Your call recording is intact and CMS-compliant — you haven't broken the chain of enrollment
- Every call ships with a structured, signed audit artifact — when CMS asks, you don't reconstruct, you export
- Subcontractor BAA status is logged per-call — you can prove which LLM provider was on the line on a given date

If a vendor in your stack has a breach, the breach exposes tokens, not PHI. Your incident response goes from "60-day breach notification to every affected beneficiary plus HHS plus state AGs" to "logged incident, no PHI exposed, no notification required."
